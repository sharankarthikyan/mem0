OpenMemory — Production Architecture

Self-hosted memory layer for AI agents · MCP server + RAG/CAG retrieval pipeline

● self-hosted ● external (OpenAI) ● planned migration
Stack: FastAPI · Qdrant (hybrid dense+BM25) · SQLite (default) / Postgres (prod) · FastEmbed ONNX reranker · Cloudflare Access edge · Railway · mem0-mcp.trybabble.io

1 · Production topology (Railway)

Everything in the data plane is co-located and self-hosted on Railway, fronted by a Cloudflare Access zero-trust edge on the public hostnames. The only hops off-box today are to OpenAI for fact-extraction and embeddings — both targeted for in-infra replacement. The relational store is SQLite by default (database.py) and Postgres in production (when DATABASE_URL points at it); the diagram shows the production topology.

2 · Write path — add_memories(text)

The only place an LLM does real reasoning: mem0 extracts atomic facts, embeds them (batch), deduplicates (MD5), and upserts. DB writes + categorization happen after the vector write; categorization is pushed off the request path. Each memory's Qdrant payload carries source_app="openmemory" + mcp_client. Passing infer=False stores text verbatim — no LLM extraction or dedup.

Fixed this session: categorization no longer blocks the write (was a synchronous LLM call inside the SQLAlchemy flush, and re-fired on every state change). It now runs in a post-commit threadpool — categorization.schedule_categorization().
Caveats (code-verified): (a) BM25 sparse is written only when the collection has a bm25 slot; older 1536-dim collections silently get dense-only. (b) The Postgres commit + categorization run inside the if 'results' in response block, but cache.invalidate(user_id) sits outside it — on a malformed/empty add response the cache is still invalidated though nothing was committed.

3 · Read path — search_memory(query) · the RAG/CAG pipeline

Called on essentially every user turn, so it is the hot path. ACL + active-state are resolved against Postgres and applied live on every call (even cache hits), so the cache can never leak paused/archived or cross-app data.

What each stage does

  1. ACL resolve (once) acl.resolve_accessible_ids — one Postgres lookup. None=all, set()=none (paused app / global deny → early return), {ids}=scoped. Replaces the old per-memory N+1.
  2. CAG cache cache.get_search_cache — exact normalized-query match only on the read path (lowercase + whitespace-collapse). An embedding-similarity fallback exists in the class but is currently unreachable: search_memory calls cache.get without the query embedding (the embedding is stored on set but never read). Stores the raw user-scoped pool; invalidated on add/update/delete. Avoids the embed + Qdrant round-trips on repeat queries.
  3. Dense retrieval vector_store.search — semantic kNN in Qdrant, filtered by payload user_id.
  4. Sparse retrieval vector_store.keyword_search — BM25 over the lemmatized query (matches the lemmatized index). Degrades to dense-only if the collection lacks the bm25 slot.
  5. RRF fusion retrieval.reciprocal_rank_fusion — rank-based merge (k=60); no score-scale normalization needed between cosine and BM25.
  6. Rerank reranker.FastEmbedReranker — local ONNX cross-encoder (Xenova/ms-marco-MiniLM-L-6-v2) re-sorts the whole fused pool. The single biggest precision lever; fully self-hosted, no torch.
  7. Active-state filter (live) acl.filter_results_by_active_state — drops paused/archived (their vectors still live in Qdrant). One indexed query scoped to candidate ids. Applied even on cache hits.
  8. ACL filter + truncate — apply the accessible set, then cut to SEARCH_LIMIT (over-fetch preserved until here so restricted apps still get a full page).
  9. Audit — one MemoryAccessLog row per returned memory.
Fixed this session (critical): the rewritten search had dropped the per-memory state==active check, leaking paused/archived memories for any app without ACL rules. Restored via a live, candidate-scoped state filter — verified by regression tests in tests/test_acl.py.

4 · CAG cache semantics

What's cached

The raw user-scoped candidate pool (post-rerank, pre-policy). Identical across a user's apps because Qdrant is filtered only by user_id.

Why it's safe

State + ACL are re-applied live after every cache read. A memory paused/revoked after caching can never surface. No cross-app leak.

Invalidation

On add / delete / delete-all and on the REST content update (PUT) — even though that PUT only touches Postgres, not the vector set. Pause/archive need no invalidation — handled by the live state filter.

Backend — default

In-process LRU + TTL, per uvicorn worker. Zero infra. OPENMEMORY_CACHE_TTL · OPENMEMORY_CACHE_MAX_ENTRIES · OPENMEMORY_CACHE_SIM_THRESHOLD · OPENMEMORY_CACHE_ENABLED

Backend — shared

Set OPENMEMORY_REDIS_URL → shared across all workers, survives restarts (rediss:// enables TLS via redis-py; Redis lookups are exact-key only). Dedicated var — not REDIS_URL, which selects a Redis vector store.

6 · Security & edge (Cloudflare Access)

Cloudflare Access edge

The public hostname mem0-mcp.trybabble.io sits behind a Cloudflare Access zero-trust gateway. Machine clients authenticate with a service token (CF-Access-Client-Id / CF-Access-Client-Secret) against a Service Auth policy (not "Allow"). A rejected/expired token gets a cross-origin 302 to the IdP login.

API-key auth opt-in

App-layer defense-in-depth behind CF. Constant-time hmac.compare_digest on /mcp+/api/ when OPENMEMORY_API_KEY is set (X-API-Key or Bearer). OPTIONS is exempt so CORS preflight passes. Off by default — enable on the public endpoint.

CORS

Env-driven OPENMEMORY_CORS_ORIGINS (default *). Credentials auto-disabled only for exactly ["*"]; any explicit allowlist auto-enables them. CORS is the outermost middleware (wraps auth), so 401s still carry CORS headers.

UI → API proxy — keeps the CF service token server-side

The dashboard browser never sees the API origin or the CF token. It calls the UI's own origin (relative /api/*); a Next.js server-side proxy (ui/app/api/[...path]/route.ts) injects the CF service-token headers and forwards to OPENMEMORY_API_ORIGIN. Same-origin from the browser ⇒ no CORS, no cross-domain CF cookie problems, and the secret stays on the server.

Why the manual redirect handling: undici strips the CF-Access-* headers on a cross-origin hop. The proxy follows same-origin 307s (FastAPI trailing-slash) re-attaching the token, but a cross-origin 302 means CF rejected the token — it returns a clear 502 JSON instead of silently relaying the login page as a 200. The UI hooks call relative URLs (not NEXT_PUBLIC_API_URL) to defeat build-time env freezing. Alternative: point OPENMEMORY_API_ORIGIN at the Railway private host (http://openmemory-mcp.railway.internal:8765) to bypass Cloudflare entirely — then no token is needed.

Network + data

Railway private networking between API↔relational DB↔Qdrant. Postgres-PITR is Railway-managed (not provisioned in-repo). Redis via rediss://. No secrets logged.

Secrets (UI service)

OPENMEMORY_API_ORIGIN, CF_ACCESS_CLIENT_ID, CF_ACCESS_CLIENT_SECRET — all server-side only, never NEXT_PUBLIC_* (those are frozen into the client bundle at build time).

7 · Self-hosted vs external — decisions

ComponentTodayDecisionWhy
Vector storeQdrant (Railway)self-hosted ✓co-located, owns hybrid dense+BM25
Metadata / ACL / auditSQLite (default) → Postgres (prod)self-hosted ✓source of truth; PITR is Railway-managed
RerankerFastEmbed ONNX cross-encoderself-hosted ✓CPU-cheap, no torch; biggest precision lever
Search cachein-proc / Redisself-hosted ✓latency + cost; Redis optional
EmbeddingsOpenAI text-embedding-3-small (1536)→ arctic-embed-m (768)more accurate (+3.8 MTEB) & fast on CPU; re-embed migration
Reasoning LLMOpenAI gpt-4o-minikeep (for now)see box below — CPU can't do both accurate & fast
Categorization LLMgpt-4o-mini (off-path)keep / disablestrict structured output; OPENMEMORY_DISABLE_CATEGORIZATION
The LLM is the one piece that can't go fully self-hosted on CPU-only Railway. mem0's add() is a single synchronous call whose base system prompt (ADDITIVE_EXTRACTION_PROMPT, ~8.4k tokens) is sent on every write, plus injected context (existing memories + recent messages) → ~8.4k+ tokens, prefill-bound. A malformed-JSON response silently yields zero memories (code-verified). Accuracy parity with gpt-4o-mini needs a 14B model (Qwen2.5-14B); a 7B underperforms. The figures that follow are external estimates, not measured in this repo: IFEval 81.0 ≈ gpt-4o-mini 80.4; a 14B on CPU ≈ ~6 min/write (Railway has no GPU); a GPU sidecar (Qwen2.5-14B-AWQ on vLLM, ~$600–800/mo or serverless) → ~1.7–4.6s/write. Chosen: keep gpt-4o-mini for reasoning until/unless a GPU host is added.

Note: EMBEDDER_PROVIDER defaults are coupled to LLM_PROVIDER (an Ollama LLM silently selects 768-dim nomic-embed-text), so set EMBEDDING_DIMS explicitly.
